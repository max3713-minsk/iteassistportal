Большой блок работ — разделим на 5 фаз и подтвердим перед сборкой. Конечная цель: понятный путь «подключение → календарь → протокол с чек‑боксами и графиками → автозагрузка в Seafile», плюс чистка наследия Zabbix и работающие заявки.

## Фаза A. Чистка Zabbix‑данных и «свежесть» метрик

Что чиним:
- При удалении подключения Zabbix `monitored_hosts` с этим `zabbix_connection_id` остаются «висеть» — путаница по принадлежности.
- В метриках хоста показывается последнее значение, даже если хост уже offline (нет связи) — пользователь думает, что данные свежие.

Изменения:
- Миграция: на `monitored_hosts.zabbix_connection_id` добавить `ON DELETE` поведение через триггер: при удалении подключения — мягкая очистка (`enabled=false`, `zabbix_host_id=NULL`) и пометка `host_group='archived'`, либо физическое удаление, если хост не привязан к оборудованию. Пользователю вернём UI «Архивные хосты» с возможностью «удалить навсегда».
- В `ZabbixConnections.tsx` — при удалении подключения предупреждение «X хостов потеряют связь, что сделать: архивировать / удалить».
- `EquipmentMonitoringMetrics.tsx`: если `lastclock` старше N (по умолчанию 5 мин) или хост в Zabbix `available=0` — показываем `N/A` + tooltip «Нет связи с {ts}». Тоже для `MonitoringHosts` и виджетов.
- В `zabbix-proxy.getHosts` уже возвращается availability — пробрасываем в карточку и используем как «жёсткий» признак offline.

## Фаза B. Переименование «Покрытие ТЗ» → «Покрытие регламентных работ»

- Таб в Monitoring переименовать.
- Компонент `TZCoverage` → `WorkScopeCoverage` (файл переименуем, экспорт сохраним для обратной совместимости).
- Источник данных «требований» теперь — таблица регламентных работ из `WorkScope`/`maintenance_tasks` (а не отдельный список ТЗ). Сопоставление: `требование регламента ↔ Zabbix item_key/шаблон` — редактируется вручную, хранится в `saved_graphs.tz_requirement_codes` (уже есть) или новой таблице `regulation_coverage_map`.
- В таблице покрытия — статус «Покрыто метрикой / Покрыто графиком / Не покрыто», ссылка на регламент.

## Фаза C. Протоколы: чек‑боксы «всё выполнено» + автографики по пунктам

- В `ProtocolDetail` добавить кнопки «Отметить все выполненными / невыполненными» + чек‑бокс на каждой строке `protocol_items` (статус done/skipped/failed).
- Под каждым пунктом — кнопка «График из Zabbix»:
  1. Автосопоставление: ищем `task ↔ zabbix item` через новую таблицу `task_metric_mapping (task_id, item_key, host_filter)` либо по `maintenance_tasks.automation_script` если уже задан key.
  2. Если автосопоставления нет — модалка выбора host+item (с поиском), сохранение в маппинг.
  3. Период по умолчанию = дата протокола ± 24 ч (или `period_start..period_end`).
  4. Рендер `GraphChart` инлайн + сохранение PNG в `protocol_items.auto_result.graph` (data‑url или путь в storage).

Миграция:
```
create table task_metric_mapping (
  id uuid pk, task_id uuid, zabbix_connection_id uuid,
  item_key text, host_filter text, created_by uuid, created_at, updated_at
);
```

## Фаза D. Заявки заказчика — стабилизация

- Проверить RLS: customer должен видеть/создавать только свои; engineer/admin — все.
- На `Tickets` для роли customer: спрятать поля назначения, оставить «Создать», список своих, статусы и комментарии.
- Уведомления о новой заявке engineer’у через уже существующий `notification-dispatch`.
- Быстрый аудит формы создания: обязательные поля, выбор сайта/оборудования из доступных текущему пользователю.
- (Деталки выявим по мере проверки — без раскапывания всего модуля.)

## Фаза E. Seafile экспорт протоколов с авто‑структурой папок

Edge function `seafile-upload` уже умеет загружать одиночные файлы. Добавим:

- Новая edge function `protocol-export-seafile`:
  Input: `protocol_id`, `include_graphs: bool`.
  Steps:
  1. Берём протокол + сайт + организацию + период + frequency.
  2. Генерим DOCX через существующий `export-protocol-docx` (на сервере воспроизведём минимальную версию, либо клиент шлёт уже готовый Blob — проще).
  3. Строим путь:
     ```
     /Протоколы/{Организация}/{Год}/{Частота:Ежедневные|Еженедельные|...}/{Период YYYY-MM-DD..YYYY-MM-DD}/
     ```
  4. В эту папку кладём `protocol.docx`, `graphs/{task_name}.png` для каждого пункта с графиком, `summary.json`.
  5. Возвращаем `viewUrl`.
- Subdir создаётся автоматически Seafile API при upload (parent_dir принимает несуществующий путь — реализуем через `/api2/repos/{id}/dir/?p=...&reloaddir=true POST operation=mkdir` рекурсивно).
- В `ProtocolDetail` кнопка «Отправить в Seafile» + индикатор «загружено N файлов».
- Маппинг частоты → русское имя папки: daily→Ежедневные, weekly→Еженедельные, monthly→Ежемесячные, quarterly→Квартальные, semi_annual→Полугодовые, on_request→ПоЗаявке.

## Что НЕ трогаем сейчас

- Календарь (Schedules) уже есть — только сверим, что ведёт на «свой» протокол. Если нет — добавим линк «Создать протокол на день».
- Глобальный редизайн, документы‑версионирование, push, PWA — не в этой итерации.

## Порядок выкатки

1. A (чистка/freshness) — короткая миграция + UI, безопасно.
2. B (переименование + источник данных).
3. C (чек‑боксы + автографики, новая таблица маппинга).
4. D (аудит заявок).
5. E (Seafile экспорт + структура папок).
